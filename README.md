##  The Problem We're Actually Solving

India's Q-Commerce boom — Zepto's 10-minute delivery promise, Blinkit's dark store model — is built on the backs of **500,000+ hyper-local delivery partners**. These workers are not employees. They earn per run, per hour, per shift. No work = no pay. Simple.

What's not simple is what happens when an external disruption hits.

Unlike food delivery, where bad weather just slows orders down, **Q-Commerce disruptions are binary:**

```
Dark Store Suspended → Worker has NO other pickup point → Income = ₹0
```

Every Q-Commerce worker is assigned to **one fixed dark store** within a 2–3km zone. That dark store is their entire income infrastructure. When Zepto suspends it due to flooding, extreme heat, or a civic shutdown — the worker doesn't get fewer runs. They get **zero runs**. Immediately. Completely. For the entire duration of the event.

**The numbers are brutal:**

| Event | Duration | Avg Income Lost | Current Compensation |
|-------|----------|----------------|---------------------|
| Flash flood (zone-level) | 4–8 hours | ₹400–₹700 | ₹0 |
| Cyclone warning suspension | 2–4 days | ₹2,000–₹4,000 | ₹0 |
| Extreme heat advisory | 6–10 hours | ₹500–₹900 | ₹0 |
| Dangerous AQI breach | 3–6 hours | ₹300–₹600 | ₹0 |
| Curfew / Section 144 | 1–3 days | ₹800–₹2,400 | ₹0 |

No bank product covers this. No government scheme addresses it. No platform compensates for it.

**RapidCover
 does. Automatically. In under 10 minutes.**

##  Parametric Triggers — The Heart of RapidCover


These are the **5 conditions** that automatically fire a payout. No human review. No claim form. Just data → validation → money.

### Trigger 1: Dark Store Zone Flood / Heavy Rain
```
Data Source  : OpenWeatherMap API (real-time precipitation, free tier)
               + IMD district advisory feed
               + Mock road condition / traffic API (waterlogging validation)
Threshold    : Rainfall > 55mm/hr sustained 30+ minutes in dark store pin code
               OR IMD issues orange/red alert for the district
Validation   : Zone polygon match confirmed
               + Zepto mock ops feed confirms zone suspension
               + Traffic mock API confirms road access disruption in zone
               + Partner GPS active in zone during trigger window
Payout       : Proportional to disruption hours × partner's hourly earning baseline
Why it works : Zepto suspends the dark store — platform confirms the loss.
               Traffic data adds independent second-layer validation.
```

### Trigger 2: Extreme Heat Advisory
```
Data Source  : OpenWeatherMap API + IMD heat wave bulletin
Threshold    : Temperature > 43°C sustained 4+ hours in zone
               OR State govt issues heat advisory restricting outdoor workers
Validation   : Zone-level temperature confirmed at partner's dark store location
               + Partner GPS active in zone at trigger time
Payout       : Full disruption window covered up to daily maximum
Why it works : Zepto has a documented policy of suspending operations during
               heat advisories — platform-confirmed, not self-reported
```

### Trigger 3: Dangerous AQI Breach
```
Data Source  : CPCB open data API + aqicn.org (free tier)
Threshold    : AQI > 400 (Severe category) in partner's zone for 3+ hours
               OR Government issues outdoor work restriction advisory
Validation   : Station-level AQI reading at nearest monitoring station to dark store
               + Cross-check with partner's zone polygon
Payout       : Hours of AQI breach × hourly earning rate
Why it works : AQI breaches are hyper-local — zone polygon model means only
               partners genuinely in the affected area receive payouts
```

### Trigger 4: Civic Shutdown / Section 144 / Bandh
```
Data Source  : Mock civic alert feed + NLP pipeline on news headlines (NewsAPI)
               + Mock traffic / road blockade data feed
Threshold    : Official curfew, Section 144, or declared bandh in delivery zone
               for 2+ hours during partner's active shift window
Validation   : Geographic boundary check covers partner's dark store zone
               + Traffic mock API confirms road blockades / access restrictions
               + Cross-reference with public authority announcements
Payout       : Full income coverage for shutdown duration
Why it works : Complete operational halt. Traffic blockade data provides
               a second independent validation layer beyond the civic alert.
```

### Trigger 5: Dark Store Force Majeure Closure
```
Data Source  : Mock Zepto / Blinkit operational status API
               + Mock traffic / road condition feed
Threshold    : Partner's assigned dark store closed > 90 minutes due to
               external event (not scheduled maintenance)
Validation   : Platform-side closure log verified (timestamped)
               + Partner had active shift slot during closure window
               + Road condition / traffic mock API confirms access disruption
Payout       : Full shift income covered
Why it works : Most Q-Commerce-specific trigger in existence — unique to RapidCover
.
               Dark store closure = total income blackout, platform-confirmed.
               Traffic data rules out scheduled closures vs. forced ones.
```

### Trigger Validation Matrix

```
Every claim passes through this pipeline in real-time:

[Trigger Detected via API]
```

##  AI/ML Integration Plan

AI is not decorative in RapidCover
. It drives the four most critical business functions: **risk profiling, pricing accuracy, fraud prevention, and disruption forecasting.**

### Model 1: Zone Risk Scorer
```
Purpose    : Assign a risk score (0–100) to every dark store zone at onboarding
Algorithm  : XGBoost Classifier
Features   :
  - 2-year IMD rainfall data per pin code
  - CPCB AQI historical averages per zone
  - NDMA flood plain maps (open data)
  - Dark store suspension history (mocked from platform data)
  - Road infrastructure score (OSM — single-access / flood-prone roads)
  - Historical traffic blockade frequency in zone
Output     : Zone Risk Score (0–100) shown to partner at onboarding with
             plain-language explanation ("Your zone has flooded 4 times in 2 years")
Training   : Public IMD + CPCB + NDMA + OSM datasets — all freely available
```

### Model 2: Dynamic Premium Engine
```
Purpose    : Personalize weekly premium for every partner every Monday
Algorithm  : Gradient Boosted Regression (scikit-learn)
Features   :
  - Zone Risk Score (from Model 1)
  - Partner's active hours per week (self-declared + GPS-verified)
  - Season index (monsoon / summer / winter)
  - Rolling 7-day AQI trend for zone
  - Road condition risk factor for zone
  - Partner's claim history (loyalty discount factor)
  - Dark store suspension frequency in last 90 days
Output     : Personalized weekly premium in ₹ with full itemized breakdown
             shown transparently in the app every Monday morning
Retraining : Weekly — new disruption events and claim data incorporated
```

### Model 3: Fraud Anomaly Detector
```
Purpose    : Score every claim for fraud likelihood before payout is released
Algorithm  : Isolation Forest (unsupervised) + deterministic rule layer
Features   :
  - GPS trajectory coherence — native mobile GPS (spoofing detection)
  - Run count during claimed disruption window (activity paradox)
  - Zone match between trigger geography and partner's registered zone
  - Claim frequency vs. historical disruption frequency for zone
  - Device fingerprint + IP clustering (collusion detection)
  - Traffic data cross-check (road conditions consistent with disruption claim?)
Output     : Fraud score 0–1
  < 0.50  → Auto-approve, payout released immediately
  0.50–0.75 → Enhanced validation (extra GPS + traffic data check)
  0.75–0.90 → Manual review queue with full anomaly breakdown
  > 0.90  → Auto-reject with partner notification and plain-language explanation
Baseline   : 60-day rolling window of partner behavior per zone
```

### Model 4: Disruption Predictor (Admin Dashboard)
```
Purpose    : Forecast next week's expected claim volume for insurers
Algorithm  : LSTM on weather + traffic time series
Features   :
  - IMD 7-day forecast per zone
  - Historical claim patterns per season
  - Current AQI trend
  - Traffic disruption pattern history (road closure frequency per zone)
  - Civic event calendar (public holidays, elections, known bandhs)
Output     : Expected payout liability (₹) for next 7 days per zone
             Shown on insurer admin dashboard every Sunday evening
             Enables proactive reserve allocation before disruptions hit
```

##  Fraud Detection Architecture

Q-Commerce delivery fraud has unique attack vectors that generic fraud models miss entirely. RapidCover
's fraud engine is built around **Zepto/Blinkit-specific patterns:**

### Attack Vector 1: GPS Spoofing
**Attempt:** Partner uses a GPS spoofer app to fake presence in a suspended zone.
**Detection:** Trajectory coherence analysis — spoofed GPS shows impossible speed jumps. Cross-validated with cell tower data (mocked). Unnatural trajectory → fraud score +0.40.

### Attack Vector 2: Activity Paradox
**Attempt:** Partner claims dark store suspension but actually completed runs during the window.
**Detection:** Run count cross-check via Zepto mock API. If `run_count > 0` during disruption window → hard reject with explanation sent to partner.

### Attack Vector 3: Zone Boundary Gaming
**Attempt:** Partner registers under a high-risk flood-prone dark store to maximize payout likelihood while operating in a safer zone.
**Detection:** 30-day GPS heatmap centroid must stay within 3km of declared dark store. Drift > 3km → automatic zone re-evaluation.

### Attack Vector 4: Duplicate Event Claiming
**Attempt:** Same disruption event claimed multiple times via app manipulation.
**Detection:** Every trigger event assigned a cryptographic event ID (`hash(zone_id + trigger_type + timestamp_bucket)`). Duplicate claim → hard reject at DB level. Immutable event log.

### Attack Vector 5: Collusion Ring
**Attempt:** Multiple fake partners from same device/address filing claims for the same event.
**Detection:** Device fingerprint + IP clustering. >3 policies sharing any 2 of (device/IP/address) → flagged. Weekly partner network graph analysis.

### Attack Vector 6: Synthetic Identity Enrollment
**Attempt:** Fabricated Zepto partner IDs used to enroll fake workers for mass claiming.
**Detection:** Zepto partner ID validated via mock platform API. Aadhaar-masked KYC + face liveness check at onboarding. One Aadhaar = one active RapidCover
 policy.

> **Double Indemnity Handling:** For partners working across multiple platforms, RapidCover
's policy is Aadhaar-linked, not platform-linked. Payouts are capped at actual income lost — never exceeding the partner's verified weekly earning baseline. We recommend IRDAI establish a shared **Gig Worker Parametric Claims Registry** — RapidCover
 is architected to plug into it from day one.
